1. 漏洞概述
================================================================

漏洞类型： SSRF (Server-Side Request Forgery) 服务端请求伪造
漏洞编号： SEC-001, SEC-002（参考 SEC_WHITEBOX_AUDIT_REPORT.md）
风险评级： 🔴 严重 (Critical)
影响范围： /fetch-url 路由 (app.py:390-421)
发现方式： 白盒源代码审计

SSRF 漏洞允许攻击者利用服务端发起的网络请求来访问本不允许
访问的内网资源。本次发现的两个 SSRF 漏洞分别针对：

  SEC-001: 未限制 URL 协议（允许 file:// 等协议）
  SEC-002: 未限制目标 IP 地址（允许内网地址）

2. 漏洞详情
================================================================

2.1 漏洞位置
────────────────────────────────────────────────────────────────

文件： app.py
路由： POST /fetch-url
原始代码行号： 390-421

2.2 漏洞代码（修复前）
────────────────────────────────────────────────────────────────

    @app.route("/fetch-url", methods=["POST"])
    def fetch_url():
        ...
        url = request.form.get("url", "")
        if url:
            req = urllib.request.Request(url)
            with urllib.request.urlopen(req, timeout=10) as response:
                ...

问题分析：
  a) 未校验协议类型 — urllib.request.urlopen() 默认支持
     http://、https://、file://、ftp://、data: 等多种协议
     
  b) 未校验目标 IP — 可访问本地回环(127.0.0.1)、私有IP段
     (10.x.x.x、172.16-31.x.x、192.168.x.x)、链路本地
     (169.254.x.x，含云元数据端点)

2.3 影响范围
────────────────────────────────────────────────────────────────

  ▸ 任意文件读取（file:///etc/passwd、file:///etc/shadow）
  ▸ 内网端口扫描（127.0.0.1:3306、10.0.0.1:22 等）
  ▸ 云元数据泄露（http://169.254.169.254/latest/meta-data/）
  ▸ 内网 Web 服务攻击（管理后台、Redis、Elasticsearch 等）

4. 修复措施
================================================================

4.1 修复方案总览
────────────────────────────────────────────────────────────────

  防护层    措施                         对应漏洞
  ───────  ────────────────────────────  ────────
  协议限制  白名单仅允许 http:// 和        SEC-001
            https://
  IP 限制  解析目标主机名到 IP，检查       SEC-002
           是否为内网/保留地址
  元数据    169.254.0.0/16 网段加入        SEC-002
  防护      私有地址黑名单

4.2 修复项 1：协议白名单
────────────────────────────────────────────────────────────────

  修复方式： 在发起请求前校验 URL 前缀

  if not url.startswith(("http://", "https://")):
      error_msg = "只允许访问 http:// 和 https:// 协议的 URL"

  拦截效果：
     file:///etc/passwd      → ❌ 拒绝
     ftp://ftp.example.com/  → ❌ 拒绝
     http://example.com/     → ✅ 允许
     https://example.com/    → ✅ 允许

4.3 修复项 2：解析域名并禁止内网地址
────────────────────────────────────────────────────────────────

  修复方式：
    1. 使用 urlparse() 从 URL 中提取 hostname
    2. 判断 hostname 是否为 IP 地址或域名
    3. 若是域名，调用 socket.gethostbyname() 解析为 IP
    4. 检查 IP 是否属于以下私有/保留网段：

        127.0.0.0/8     本地回环
        10.0.0.0/8      A 类私有地址
        172.16.0.0/12   B 类私有地址
        192.168.0.0/16  C 类私有地址
        169.254.0.0/16  链路本地（含云元数据）
        0.0.0.0/8       零地址
        ::1/128         IPv6 回环
        fc00::/7         IPv6 唯一本地地址

  拦截效果：
     http://127.0.0.1:8083/          → ❌ 拒绝
     http://192.168.1.1/             → ❌ 拒绝
     http://10.0.0.1/                → ❌ 拒绝
     http://169.254.169.254/latest/  → ❌ 拒绝
     http://httpbin.org/get          → ✅ 允许

4.4 新增 import
────────────────────────────────────────────────────────────────

  + import socket
  + from urllib.parse import urlparse
  + from ipaddress import ip_address, ip_network

4.5 补充：HTTP User-Agent
────────────────────────────────────────────────────────────────

  为 urllib.request 添加 User-Agent 请求头，避免被部分
  Web 服务因缺少 User-Agent 而拒绝。

        except urllib.error.URLError as e:
            ...
        except Exception as e:
            ...

    ⚠️ 直接使用用户输入的 URL，无任何校验

5.2 修复后
────────────────────────────────────────────────────────────────

    url = request.form.get("url", "")

    if url:
        # SSRF 防护1：只允许 http/https 协议
        if not url.startswith(("http://", "https://")):
            error_msg = "只允许访问 http:// 和 https:// 协议的 URL"
        else:
            try:
                parsed = urlparse(url)
                hostname = parsed.hostname

                # SSRF 防护2：解析域名并检查是否为内网
                try:
                    addr = ip_address(hostname)
                except ValueError:
                    addr = ip_address(socket.gethostbyname(hostname))

                # SSRF 防护3：禁止内网/保留地址
                private_networks = [
                    ip_network("127.0.0.0/8"),
                    ip_network("10.0.0.0/8"),
                    ip_network("172.16.0.0/12"),
                    ip_network("192.168.0.0/16"),
                    ip_network("169.254.0.0/16"),
                    ip_network("0.0.0.0/8"),
                    ip_network("::1/128"),
                    ip_network("fc00::/7"),
                ]
                if any(addr in net for net in private_networks):
                    error_msg = "禁止访问内网或保留地址"

            except Exception as e:
                error_msg = f"URL 校验失败: {str(e)}"

    ✅ 协议白名单 + IP 黑名单双重防护

7. 遗留风险
================================================================

7.1 DNS 重绑定攻击
────────────────────────────────────────────────────────────────

  风险： 攻击者使用一个域名，第一次解析返回公网 IP（通过校验），
         随后快速切换 DNS 记录指向内网 IP，使第二次请求打到内网。

  缓解： 当前修复在请求前解析一次 IP，但仍存在 TOCTOU 窗口。
  建议： 增加 urllib 级别的 DNS 解析回调，在建立连接时二次校验。

         # 进阶方案：使用 requests + 自定义 transport 或
         # 在 socket 层 hook getaddrinfo 进行双重校验

7.2 重定向跳转
────────────────────────────────────────────────────────────────

  风险： 合法的外网 URL 可能返回 302 重定向到内网地址
         例: http://attacker.com/redirect?to=http://127.0.0.1:8083/

  建议： 设置 urllib.request.HTTPRedirectHandler，在重定向时
         重新校验目标 URL

         class SSRFRedirectHandler(urllib.request.HTTPRedirectHandler):
             def redirect_request(self, req, fp, code, msg, headers, newurl):
                 # 对 newurl 重新执行 SSRF 校验
                 validate_url(newurl)
                 return super().redirect_request(req, fp, code, msg, headers, newurl)

7.3 URL 解析差异
────────────────────────────────────────────────────────────────

  风险： urllib.parse.urlparse 与 urllib.request.urlopen 对某些
         畸形 URL 的解析可能存在差异，导致绕过

  建议： 使用权威解析库或白名单方式校验
